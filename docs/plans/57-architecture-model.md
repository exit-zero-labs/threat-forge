# Issue 57 — Define the backward-compatible system architecture model

## Objective

A `.thf` document can describe general system architecture — layers, non-security groups,
non-data-flow relationships, component classification, and typed views — and can carry that data
losslessly through every existing read, edit, and write path, while every file written before this
change still parses and re-serializes byte-for-byte unchanged.

### Deferred follow-ups (file as separate issues; do not build here)

1. **Per-view layout.** `diagrams` is already a `Vec`, but `Element.position` is a single inline
   position (ADR-006). True multi-view requires per-view positions, which is a structural change to
   the layout model, not an additive field. Today the app reads only `model.diagrams[0]`
   (`src/lib/model-layout-utils.ts:8`). Multi-view rendering and per-view layout belong in one
   dedicated issue after `#58` defines the view UX.
2. **Free-form entity attributes** (`attributes: map<string,string>`). Rust `BTreeMap` emits
   sorted keys; a JS object emits insertion order. Introducing maps now would add a new
   cross-language ordering divergence on top of the ones `#94` already has to reconcile.
3. **`line_style` / `collapsed` and other new visual vocabulary.** This issue only mirrors visual
   fields that already have precedent on `DataFlow` and `TrustBoundary`. New visual concepts should
   land with their renderer in `#58`.
4. **Threats targeting groups or relationships.** Threat targets stay `element` / `flow`.
5. **Widening `validate_version` to accept any `1.x`.** Analyzed and rejected below.

## Current behavior and evidence

All statements here were verified against the working tree at `e6a721d`.

### Schema shape

The Rust model in `src-tauri/src/models/threat_model.rs` is the schema of record. `ThreatModel`
declares `version`, `metadata`, `elements`, `data_flows`, `trust_boundaries`, `threats`, `diagrams`
(lines 7–20). All five collection sections carry `#[serde(default)]` but **no**
`skip_serializing_if`, so they are always emitted, even when empty. Optional scalar and layout
fields consistently use `#[serde(default, skip_serializing_if = "Option::is_none")]`, and
`Element.technologies` uses `#[serde(default, skip_serializing_if = "Vec::is_empty")]` (line 100).
There is no `deny_unknown_fields` anywhere, and `test_unknown_fields_are_tolerated`
(line 539) locks that in.

`src/types/threat-model.ts` is a hand-maintained mirror. It has no generator and no test that
compares it to the Rust model, so the two can drift silently today. `ElementType` is
`string` (line 7), i.e. the component-type vocabulary is already open and needs no schema change to
absorb the `#59` registry.

### Serialization paths

- **Desktop:** `src-tauri/src/file_io/writer.rs::write_threat_model` prepends
  `# ThreatForge Threat Model\n# Generated by ThreatForge v{CARGO_PKG_VERSION}\n` and then
  `serde_yaml::to_string`. The version is interpolated from the crate version, so a byte-comparison
  against a whole written file would break on every release bump.
- **Desktop read:** `src-tauri/src/file_io/reader.rs::read_threat_model` parses, then
  `validate_version` (exact match on `"1.0"`, line 104) then `validate_references` (line 115).
- **Browser:** `src/lib/adapters/browser-file-adapter.ts` uses `js-yaml` 4.3.0 —
  `yaml.load` on open with `??=` normalization for the five collection sections and
  `element.technologies`, and `yaml.dump(model, { lineWidth: -1, noRefs: true, sortKeys: false,
  quotingType: '"' })` on save.

### Measured emitter behavior (probes run against the pinned dependency versions)

`serde_yaml` 0.9 for a model equivalent to the browser one emits:

```yaml
version: '1.0'
metadata:
  created: 2026-03-15
  description: ''
elements:
- id: a
  position:
    x: 100.0
```

`js-yaml` 4.3.0 with the adapter's exact options emits:

```yaml
version: "1.0"
metadata:
  created: "2026-03-15"
  description: ""
elements:
  - id: a
    position:
      x: 100
      "y": 200.5
```

Four independent divergences exist **today, before this change**: quote style (`'` vs `"`), date
quoting, sequence indentation (0 vs 2), and float formatting (`100.0` vs `100`). js-yaml also quotes
the key `y` because YAML 1.1 resolves bare `y` as a boolean.

Two consequences follow, and this plan depends on both:

1. **Byte-identical output between the Rust writer and the browser writer is not achievable
   today.** Any contract test must assert Rust-side byte identity plus cross-language *structural
   and key-order* identity, not cross-language byte identity. `#94`'s acceptance criterion
   "browser-written and Rust-written output for the same model remain byte-identical to each other"
   is not satisfiable by a `js-yaml` version bump alone; it needs a normalizing emitter or an
   amended criterion. Flag this to the owner rather than assuming it.
2. **`skip_serializing_if` is the diff-minimality mechanism, and it is verified.** In the probe, a
   `Vec` carrying `skip_serializing_if = "Vec::is_empty"` was omitted entirely while a `Vec` without
   it emitted `threats: []`. Serde also emitted every struct field in declaration order. So
   declaration order *is* canonical field order, and `skip_serializing_if` on every new field is
   what keeps a pre-existing file's bytes unchanged.

### Reference integrity today

`validate_references` (reader.rs:115) rejects duplicate `elements` IDs, duplicate `data_flows` IDs,
`data_flows.from`/`to` that name no element, `trust_boundaries.contains` entries that name no
element, and `threats.element`/`threats.flow` that name nothing. It does **not** check for ID
collisions *between* sections. An existing file may legally contain an element and a trust boundary
sharing one ID, so adding a cross-section collision rule over existing sections would reject
previously valid files. New-section collision rules are safe because no existing file can contain a
new section.

### Why relationships cannot be folded into `data_flows`

`src-tauri/src/stride/mod.rs:257` iterates `for flow in &model.data_flows` and applies every rule
whose `targets_flows` is true (three rules, lines 152/160/168). `src/lib/stride-engine.ts` mirrors
this. Modeling `deploys_to` or `depends_on` edges as data flows would therefore manufacture
tampering, information-disclosure, and denial-of-service threats on non-data edges — a
false-positive engine, directly against the `threat-model-expert` lane's charter. A separate
`relationships` section is a correctness requirement, not a stylistic preference.

### Pass-through survivability of new top-level sections

Verified by reading the mutation paths: `src/stores/model-store.ts` builds every new state with
`{ ...model, <section>: updated }`; `src/hooks/use-file-operations.ts::captureCanvasIntoModel`
(line 33) returns `{ ...model, elements, trust_boundaries, data_flows, diagrams }`;
`src/stores/history-store.ts` uses `structuredClone`; `src/stores/clipboard-store.ts` copies
entities with `{ ...element, ... }` / `{ ...boundary, ... }`. All of these preserve unlisted
properties. **No store or hook change is required for new top-level sections or new element fields
to survive an open → edit → save cycle.** This is what keeps this issue out of `#53`'s files.

The asymmetric cost is on the Rust side: Rust struct literals must name every field, so each new
`Element` field forces an edit at every literal construction site. Those sites are
`src-tauri/src/importers/tm7.rs:155`, `src-tauri/src/mcp/server.rs:303`,
`src-tauri/src/ai/prompt.rs:229`, and `src-tauri/src/stride/mod.rs` lines 328, 347, 366, 678. The
compiler enforces exhaustiveness there, which is a feature — do not paper over it with
`..Default::default()`.

### Test and fixture infrastructure

There is no `tests/` directory and **no `.thf` fixture exists anywhere in the repository**, despite
`docs/knowledge/file-format.md:218` already promising "Sample `.thf` files in `tests/fixtures/`".

Vitest is configured with `include: ["src/**/*.test.{ts,tsx}"]` and `environment: "jsdom"`

does not exclude `tests/`, so a fixture corpus is committable and visible to Docker CI.

### 2. Decide and record the schema version policy (ADR-009)

- **Behavior:** `version` stays `"1.0"`. `validate_version` keeps its exact-match, fail-closed
  behavior and is not modified. The decision, its alternatives, and its residual risk are written
  down.
- **Files:** `docs/knowledge/architecture.md` (append ADR-009 row),
  `docs/knowledge/file-format.md` (new "Schema versioning policy" section).
- **Implementation:** Transcribe the following decision. It is settled; do not re-litigate it during
  implementation, and do not hand it back to the owner as an open question.

  **Decision: no version bump. `.thf` documents produced by this change remain `version: "1.0"`.**

  Reasoning:
  1. *The change is additive by construction.* No field is removed, renamed, retyped, or given new
     semantics. `AGENTS.md` requires a version bump for **breaking** schema changes; this is not
     one, and there is nothing to migrate.
  2. *A bump would convert graceful degradation into a hard failure.* `validate_version`
     (`reader.rs:104`) matches `"1.0"` exactly and returns `UnsupportedVersion` for anything else.
     Because serde tolerates unknown fields (proven by `test_unknown_fields_are_tolerated`), an
     already-released 0.2.x build can open a file containing `layers`, `groups`, and `relationships`
     and simply ignore them. Bumping to `"1.1"` would make every already-shipped build refuse to
     open **any** file saved by the new version — including plain threat models with no architecture
     data at all. That trades a narrow failure mode for a universal one and directly attacks the
     "portable" product invariant.
  3. *The version field's job is compatibility signalling, not feature advertisement.* Presence of
     the `layers` / `groups` / `relationships` sections is the feature signal, and it is
     self-describing.

  Residual risk, which must be documented rather than hidden: if an older build opens a document
  containing architecture sections and saves it, serde discards the unknown sections and the
  architecture data is lost. `.github/instructions/file-formats.instructions.md` forbids claiming
  generic preservation, so state this explicitly in `docs/knowledge/file-format.md` instead of
  implying round-trip safety across versions.

  Alternatives considered and rejected:
  - *Bump to `"1.1"`.* Rejected per (2): it makes downgrade a total failure instead of a partial
    one, for a change that has no breaking element.
  - *Keep `"1.0"` but widen `validate_version` to accept any `1.x`.* Rejected. It cannot help the
    installed base — already-released builds hard-fail regardless — and going forward it would
    convert a clear refusal into silent data loss when a future build meets an unknown `1.x` file.
    Exact-match is the fail-closed behavior and stays.

  Also record the forward rule: **additive optional fields never bump the version; a bump happens
  only for a breaking change, and a breaking bump is expected to hard-fail on older builds.**
- **Targeted verification:** `grep` confirms `validate_version` is unchanged and no `"1.1"` literal
  is introduced. The existing `test_rejects_unsupported_version` still passes.
- **Intent validation:** Owner accepts the downgrade-data-loss tradeoff, or directs a bump. This is
  the single highest-leverage decision in the issue.

### 3. Add the architecture sections and fields to the Rust model

- **Behavior:** `ThreatModel` gains `layers`, `groups`, and `relationships`; `Element` gains
  `layer`, `group`, and `tags`; `Metadata` gains `threat_analysis_enabled`; `Diagram` gains `kind`
  and `description`. Every addition is optional and omitted when unset.
- **Files:** `src-tauri/src/models/threat_model.rs`, plus the seven `Element` literal sites listed in
  the evidence section (`importers/tm7.rs`, `mcp/server.rs`, `ai/prompt.rs`, `stride/mod.rs` ×4).
- **Implementation:**
  1. Canonical section order becomes declarations-before-references:
     `version → metadata → layers → groups → elements → data_flows → relationships →
     trust_boundaries → threats → diagrams`. Do not reorder any existing field or section relative
     to the others; only insert. Because the three new sections carry
     `skip_serializing_if = "Vec::is_empty"`, insertion position cannot perturb any existing file.
  2. New root sections, each
     `#[serde(default, skip_serializing_if = "Vec::is_empty")] pub …: Vec<…>`:
     - `Layer { id: String, name: String, description: Option<String> }`. Array order **is** display
       order; do not add an `order` integer, which would create renumbering churn on every reorder.
     - `Group { id, name, r#type: Option<String>, parent: Option<String>, description: Option<String>,
       position: Option<Position>, size: Option<Size>, fill_color, stroke_color, fill_opacity,
       stroke_opacity }`. Field set and serde attributes mirror `TrustBoundary` exactly so `#58` can
       reuse the boundary node machinery. Serialize `r#type` as `type` with `#[serde(rename = "type")]`
       following the `Element::element_type` precedent.
     - `Relationship { id: String, #[serde(rename = "type")] relationship_type: String, from: String,
       to: String, name: Option<String>, description: Option<String>, source_handle, target_handle,
       label_offset: Option<Position>, stroke_color, stroke_opacity }`. `type` is **required** (no
       `#[serde(default)]`): the section is new, so requiring it costs no backward compatibility and
       fails closed on an untyped relationship. The visual fields deliberately mirror `DataFlow` and
       nothing more; no new visual vocabulary here.
  3. `Element` additions, placed for readability: `layer: Option<String>` and `group: Option<String>`
     immediately after `trust_zone`; `tags: Vec<String>` immediately after `technologies` with the
     identical `#[serde(default, skip_serializing_if = "Vec::is_empty")]`.
  4. Membership lives on the element (`element.layer`, `element.group`), **not** as `contains` lists
     on the container. Rationale to record in the doc update: one source of truth, and
     reclassifying a component is a single changed line in a single hunk rather than two hunks
     across two container blocks. `trust_boundaries.contains` is unchanged existing behavior; note
     the asymmetry in `docs/knowledge/file-format.md` so it does not read as an accident.
  5. `Metadata.threat_analysis_enabled: Option<bool>` with
     `#[serde(default, skip_serializing_if = "Option::is_none")]`. Tri-state is required: `None`
     must not mean "disabled", or every existing file would appear to have threat analysis off.
  6. `Diagram.kind: Option<String>` and `Diagram.description: Option<String>`. `kind` is a free-form
     string (`architecture`, `dfd`, `deployment`, …) following the `ElementType` precedent, so `#59`
     owns the vocabulary and no schema change is needed to extend it.
  7. Update `ThreatModel::new` to initialize the three new vectors as empty and the new options as
     `None`. Update the seven `Element` literal sites by naming the new fields explicitly. Do **not**
     derive `Default` for `Element` and do **not** use `..Default::default()`: the compiler's
     exhaustiveness error at each site is the drift guard.
- **Targeted verification:**
  `cargo test --manifest-path src-tauri/Cargo.toml` — every Step 1 canonical byte assertion must
  still pass **unmodified**. Add `test_new_sections_omitted_when_empty` asserting that
  `serde_yaml::to_string(&ThreatModel::new(…))` contains none of `layers`, `groups`,
  `relationships`, `tags`, `threat_analysis_enabled`, `kind`.
- **Intent validation:** Owner reviews the new YAML shape in the Step 6 fixture and confirms it
  reads naturally for someone describing a system rather than a threat model.

### 4. Extend reference-integrity validation for the new sections

- **Behavior:** A `.thf` file with an inconsistent architecture graph is rejected on read with an
  actionable error. No previously valid file becomes invalid.
- **Files:** `src-tauri/src/file_io/reader.rs`, `src-tauri/src/errors.rs`,
  `tests/fixtures/thf/invalid/*.thf`.
- **Implementation:**
  1. Extend `validate_references` with duplicate-ID checks for `layers`, `groups`, and
     `relationships`, reusing `ThreatForgeError::DuplicateId` with the section name.
  2. Reference checks, all reusing `ThreatForgeError::InvalidReference`:
     `element.layer` → `layers[].id`; `element.group` → `groups[].id`; `group.parent` →
     `groups[].id`; `relationship.from`/`to` → `elements[].id`. Relationship endpoints are element
     IDs only, matching `data_flows`; record that limitation in the format doc.
  3. Cycle detection for `group.parent`, including self-parent. Add one new error variant:
     `#[error("Circular group nesting detected at group '{id}'")] CircularGroupNesting { id: String }`.
     Iterative traversal with a visited set; no recursion, no `unwrap`.
  4. Namespace-collision checks that are safe by construction because they only constrain new
     sections: `groups[].id` must not collide with any `elements[].id` or `trust_boundaries[].id`
     (they share the ReactFlow node ID space — `clipboard-store.ts` maps all three into one
     `idMap`); `relationships[].id` must not collide with any `data_flows[].id` (shared edge ID
     space). **Do not** add a collision rule between two pre-existing sections: an
     element and a boundary may legally share an ID in files that already exist, and adding that
     rule would reject them.
  5. Add one `invalid/` fixture per new rule and assert the specific error variant, not just
     `is_err()`.
- **Targeted verification:**
  `cargo test --manifest-path src-tauri/Cargo.toml file_io::reader`. Every Step 1 valid fixture must
  still read successfully — that is the proof no existing file was invalidated.
- **Intent validation:** Owner reads one rejection message and confirms it names the offending ID
  and field clearly enough to fix by hand, and that it leaks no path or internal detail.

## Cross-cutting requirements

- **Security and privacy:** The trust boundary is `read_threat_model` — `.thf` content is untrusted
  input. Every new reference is validated before use, group nesting is cycle-checked so no traversal
  can loop, and traversal is iterative with no `unwrap`. `relationship.type` is display metadata and
  must never gate a security decision. No new file, network, or IPC surface is introduced; the
  `ThreatModel` IPC payload simply carries additional optional fields on the existing
  `open_threat_model` / `save_threat_model` commands. No secrets are involved. Rejection errors name
  user data (IDs, field names) and never paths or raw serde internals.
- **`.thf` compatibility:** No version bump (ADR-009). Every new field is optional with
  `skip_serializing_if`, so pre-change files re-serialize byte-for-byte identically — asserted, not
  assumed, by the canonical fixtures committed in Step 1 and re-run unmodified after Step 3.
  Round-trip, old-file, golden, malformed-input, and legacy-sidecar-migration coverage are all
  required by `.github/instructions/file-formats.instructions.md` and are enumerated in Steps 1, 4,
  and 6. Unknown fields remain tolerated; `test_unknown_fields_are_tolerated` must not be weakened.
- **Browser and desktop:** Both paths must accept and preserve the new sections. The difference is
  deliberate and pre-existing: Rust is the canonical writer and the byte-identity target; the
  browser writer is asserted on structure and key order only, because the two emitters already
  diverge on quoting, sequence indentation, date quoting, and float formatting. Record that
  divergence for `#94` rather than silently accepting or "fixing" it here.
- **AI safety:** No AI path changes. `src-tauri/src/ai/prompt.rs` and `src/lib/ai-action-executor.ts`
  gain no ability to write the new sections in this issue, so no new untrusted-mutation surface is
  created. If a reviewer asks for AI authoring of layers/groups, that is a separate issue behind the
  existing validation and undo machinery.
- **Accessibility and UX:** No UI surface changes. The only user-visible behavior is a file that
  fails to open with a clear message when its architecture graph is inconsistent.
- **Observability and evidence:** The PR should include the full text of
  `architecture-canonical-full.thf`, the output of the minimal-edit diff test, and an explicit
  statement that every Step 1 assertion passed unmodified after Steps 3 and 4.

## Owner validation

Deterministic checks cannot decide any of the following:

1. **Is the taxonomy the right one?** `layers` + `groups` + element-side membership +
   free-form `relationship.type` is a modeling opinion. The plausible-but-wrong outcome is a schema
   that passes every test and that nobody wants to write by hand. Read
   `architecture-canonical-full.thf` end to end and judge it as a document, not as a data structure.
2. **Is the no-bump decision acceptable?** The accepted cost is that an older ThreatForge build can
   open an architecture document, save it, and silently discard `layers`, `groups`,
   `relationships`, and element `layer`/`group`/`tags`. Confirm that is preferable to older builds
   refusing to open every new file.
3. **Is element-side membership right?** It diverges from `trust_boundaries.contains`. Confirm the
   asymmetry is worth the one-line-diff benefit, or direct a consistent `contains` design.
4. **Is the three-artifact contract friction acceptable?** Every future schema field must be added
   to the Rust helper, the TypeScript literal, and the fixture together. That is intentional
   friction, and it is the only mechanism preventing silent TS/Rust drift.
5. **Is shipping schema ahead of UI acceptable?** `layers`, `groups`, and `relationships` have no
   authoring surface until `#58`/`#59`. The mitigation is that the persistence contract is
   end-to-end tested and the sections are consumed by `#58`, `#59`, and `#60`. Confirm the sequencing,
   or direct that `#57` merge only alongside a consumer.

   gains a normalizing emitter or that criterion is amended.
